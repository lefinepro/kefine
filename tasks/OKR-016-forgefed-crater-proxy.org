#+TITLE: OKR-016: ForgeFed Crater Proxy Service
#+DATE: 2026-02-21
#+STARTUP: overview

* Overview
Create "Crater" - a stateless proxy service written in **Crystal** that converts the simple task API to **ActivityPub + ForgeFed** format. This separates concerns: the main application uses a simple REST API, while Crater handles all federation complexity as a translation layer.

* Why Crystal?
- **Performance**: Near Ruby speed with C-like efficiency
- **Concurrency**: Fiber-based lightweight concurrency
- **Type safety**: Compile-time type checking
- **Syntax**: Clean, Ruby-like syntax
- **HTTP built-in**: Excellent HTTP client/server in stdlib
- **JSON handling**: Native JSON support with types
- **Small binaries**: Easy deployment
- **No runtime**: Single static binary

* ForgeFed Support
Crater implements full **ForgeFed** protocol on top of ActivityPub:
- **Project** type (federated project metadata)
- **Membership** management (Add/Remove activities)
- **Project Follow** (Follow/Accept activities)
- **Repository** linking
- **Issue Tracker** integration

* Ticket Types
Crater supports multiple ticket types for ForgeFed:
- **Bug Report**: Issue with reproduction steps, severity
- **Feature Request**: New feature proposal with description
- **Support Ticket**: User support request
- **Task**: General work item
- **Review**: Code review or documentation review

* Architecture Layers

** Layer 1: Proxy Service Layer (Direct Access)
- Kefine App connects directly to proxy services
- Crater handles all federation complexity
- Future proxies: Auth, Sync, Analytics, etc.
- =Stateless services, no database=

** Layer 2: Data API Layer (Simple API)
- Simple REST API for CRUD operations
- Used by Crater (and other proxies) to persist data
- Business logic and validation
- =Stateless service=

** Layer 3: Local Store (Primary Database)
- Local database service with persistence
- Manages task/project/user state
- Can be PostgreSQL, SQLite, or any database
- =Primary data store=

** Layer 4: External Federation Servers (Distributed Store)
- ForgeFed servers store federated data
- ActivityPub servers store federated activities
- Data is replicated across federation
- =Distributed storage via federation protocol=

* Key Principles
- **Direct proxy access**: Kefine App connects directly to proxy services
- **Crater is stateless**: No database, no persistence, pure translation
- **Simple API is internal**: Used by proxies, not directly by frontend
- **Dual storage**: Local store + External federation servers
- **Federation as storage**: External servers act as distributed data store
- **Independent deployment**: Each layer can scale/deploy separately
- **Crater is optional**: Main app can work without federation proxy

* Scope

** In Scope (Crater Service)
- ForgeFed ActivityPub protocol implementation
- Task ↔ Activity mapping (Create, Update, Delete, Comment)
- Actor representation (User ↔ Actor)
- Project ↔ Collection/OrderedCollection mapping
- Signature verification for incoming requests
- HTTP signature for outgoing requests
- WebFinger endpoint support
- NodeInfo endpoint support
- Inbox/Outbox handling

** In Scope (Main App Integration)
- Simple task API (REST or GraphQL)
- Basic CRUD operations
- User authentication (Reown AppKit from OKR-015)
- Optional Crater connection configuration

** Out of Scope
- Direct ForgeFed integration in main app
- Complex federation logic in frontend
- State management for federation in main app

** Kemal Server Example (src/crater.cr) - Pure API
#+BEGIN_SRC crystal
require "kemal"
require "./crater/**"

# WebFinger endpoint - pure JSON API
get "/.well-known/webfinger" do |env|
  env.response.content_type = "application/jrd+json"
  WebFingerHandler.handle(env).to_json
end

# Actor endpoint - ActivityPub JSON
get "/actor" do |env|
  env.response.content_type = "application/activity+json"
  ActorHandler.handle(env).to_json
end

# Inbox endpoint - receives ActivityPub JSON
post "/inbox" do |env|
  body = env.request.body.gets_to_end
  activity = Activity.from_json(body)
  InboxHandler.handle(activity)
  env.response.status_code = 202
  {"status": "accepted"}.to_json
end

# Outbox endpoint - pure JSON collection
get "/outbox" do |env|
  env.response.content_type = "application/activity+json"
  OutboxHandler.handle(env).to_json
end

# Health check endpoint
get "/health" do |env|
  env.response.content_type = "application/json"
  {"status": "ok", "service": "crater"}.to_json
end

# Start Kemal server (API only, no templates)
Kemal.run
#+END_SRC

* Deployment

** Main App
- Deploy to Vercel, Netlify, or any static host
- API can be serverless functions or separate service

** Crater Service (Crystal)
- Build static binary: ~crystal build src/crater.cr --release --static~
- Deploy binary to:
  - Railway (Docker with Crystal base image)
  - Render (Docker)
  - Fly.io (native binary)
  - AWS Lambda (custom runtime)
- Requires persistent domain for federation
- SSL certificate mandatory (HTTPS via reverse proxy)
- Stateless - can scale horizontally
- Docker image size: ~10-20MB (static binary)

** Docker Example
#+BEGIN_SRC dockerfile
FROM crystallang/crystal:1.11.0-alpine AS builder
WORKDIR /app
COPY shard.yml shard.lock ./
RUN shards install
COPY src ./src
RUN crystal build src/crater.cr -o crater --release

FROM alpine:latest
RUN apk --no-cache add ca-certificates
WORKDIR /root/
COPY --from=builder /app/crater .
EXPOSE 3001
CMD ["./crater"]
#+END_SRC

* Risks & Mitigation

| Risk | Impact | Mitigation |
|------+--------+------------|
| ActivityPub complexity | High | Use libraries, follow specs strictly |
| HTTP signature issues | High | Test with known-good implementations |
| Federation delivery failures | Medium | Implement retry queue, logging |
| Security vulnerabilities | High | Validate all inputs, use HTTPS |
| Performance bottlenecks | Medium | Benchmark early, optimize hot paths |
| Crater service downtime | Low | Main app works independently |
